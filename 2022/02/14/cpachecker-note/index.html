Note of CPAChecker

Created2022-02-14Updated2022-02-14

This is the note of paper “CPAchecker: A Tool for Configurable Software Verification” by Dirk Beyer and M. Erkan Keremoglu at CAV’11.

Overview

It is hard to understand the essential insights of new approaches of software verification.

  • It takes a considerable engineering effort to build the software infrastructure for evaluating verification algorithms.
  • Also, successful approaches in program analysis and model checking are rarely combined.
  • Moreover, most published approaches are not comparable due to the impact of different environment in different implementation.

Configurable program analysis (CPA) has a conceptual basis for expressing different verification approaches in the same formal setting, by providing an interface for the definition of program analysis such as abstract domain, post operator, merge operator and stop operator. With CPAChecker, the comparison of different approaches in the same experimental setting becomes easy and the experimental results will be more meaningful.

Architecture

CPAChecker's architecture is shown in Figure 1 below. Its central data structure is a set of control-flow automata (CFA) which consists of control-flow locations and control-flow edges. Here, a location represents a program-counter value, and an edge represents a program operation (an assume operation, an assignment block, a function call or a function return). This framework also provides interfaces to SMT solvers and interpolation procedures.

Figure 1: Architecture of CPAChecker

In Figure 1, the core part of CPAChecker is the CPA algorithm, which performs the reachability analysis and operates on an object of the abstract data type CPA.

In order to extend CPAChecker by integrating an additional CPA for a new abstract domain, only two steps are necessary:

  1. An entry in the global properties file to announce the new CPA for composition
  2. The implementations of all CPA needs to be provided

The interaction is shown in Figure 2. The CPA algorithm takes as input a set of CFA representing the program and a (composite) CPA. The elements in the gray box (top right in Figure 2) represent the abstract interfaces of the CPA and the CPA operations.

Figure 2: Interaction among CPAs

Evaluation

The authors compare CPAChecker with several existing tools. They want to show the CPAChceker can

  • contributes a great flexibility by its configuration possibilities, and
  • improves the performance due to the interesting analysis configuration

The authors use three sets of benchmarks: simplified Windows device drivers, simplified state machine of SSH-suite and SystemC programs.

The verification results are shown in Table 1.

Table 1: Evaluation Results

In Table 1, the first column reports the program name and the second column indicates the expected verification result; The other 4 columns show the result of 4 checkers. For each result, runtime is given in seconds of CPU time; TOTOTO and MOMOMO means the run was terminated since it ran out of time (900s) and out of memory (12GB); ✓\checkmark✓ indicates that the expected result was correctly computed; ERERER indicates that the checker failed to return a verification result.

Here is the performance of those 4 verification tools(CBMC, SAT_ABS, BLAST and CPAChecker):

  • CBMC uses bounded model checker (loop bound is set as 10) and it has detected many of bugs and only proved 5 safety cases. In general, nothing can be said about safe programs.
  • SAT_ABS performs an explicit CEGAR loop.
  • BLAST is configured to employ lazy, interpolation-based refinement, DFS for state-space exploration and the recommended predicate-search heuristic.
  • CPAChecker has 2 configuration options: predicate analysis (on the left) and explicit-value analysis (on the right).

Summary

In general, CPAChecker's predicate analysis outperforms other tools. Also, for the programs with a bug, CPAChecker can be started with an explicit-value analysis and compute the result within seconds.
