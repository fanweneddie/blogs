Combining Model Checking with Data-Flow Analysis

Created2022-02-17Updated2022-02-17

This is the note of Chapter 16: “Combining Model Checking and Data-Flow Analysis” in “Handbook of Model Checking”.

The chapter presents combined verification techniques in the framework of configurable program analysis, in order to emphasize techniques used in data-flow analysis and in model checking.

Introduction

For software verification, model checking is a semi-decidable, exhaustive and precise analysis of an abstract model of a program; Whereas data-flow analysis is a terminating, imprecise abstract interpretation of a concrete model of a program.

However, this chapter will illustrate that those two approaches are theoretically equivalent. Here we go!!!

General Considerations

We compare three techniques of program validation: type checking, data-flow analysis and model checking. They have different representation of program, property and analysis algorithm.

Type Checking

Type checking is an analysis of a program’s syntax tree where the property of types is embedded. It can validate the static type-safety properties and can help compiler generate a target code.

  • The program representation is a syntax tree.
  • The property representation is the type restriction in the syntax (can be propositional logic or predicate logic).
  • The analysis algorithm is a finite tree-traversal algorithm.

Data-Flow Analysis

Data flow analysis predicts the flow of information through the locations of a program. Since the analysis over-approximates a program’s possible execution sequences, its results are imprecise. It can be used for generating efficient code.

  • The program representation is a control flow automaton (CFA), where a node represents a program location and an edge represents a primitive action that a transfer controls. (CFA is a little different from CFG, where a node represents an action and an edge represents a data/control flow)

  • The property representation is a set or map that indicates the variables or expressions whose values where transferred along a control path to the program location.

  • The analysis algorithm is an iterative algorithm (least-fixed-point for forward and greatest-fixed point for backward) that propagates the properties along the CFA’s edges. Here, the semantic of each edge is defined by a transfer function. There are two important versions of the analysis algorithm:

    • Maximal Fixed Point (MFP): Properties for each program location are computed directly on the CFA, and the information computed for each program location l′l'l′ is defined by the following equation.

      propertyAtNode(l′)=⋃l∈pred(l′)fl→l′(propertyAtNode(l))propertyAtNode(l') = \bigcup_{l \in pred(l')} f_{l \rightarrow l'}(propertyAtNode(l)) propertyAtNode(l′)=l∈pred(l′)⋃​fl→l′​(propertyAtNode(l))

    • Meet Over All Paths (MOP): The iterative analysis algorithm enumerates the paths within the CFA. Properties are computed for all program locations on each individual path, and the results of all the analyses on all paths are joined.

    Usually, MOP analysis can be more precise than MFP analysis, but their results may coincide if the transfer functions distribute over meet operator ⊔\sqcup⊔.

Model Checking

Model checking enumerates the sequences of states that arise during a program’s execution and decides whether the sequences of states satisfy a safety property.

  • The program representation is a directed graph whose nodes are the program’s run-time states, connected by edges that define sequencing. Since the graph may be infinite, we need to represent it by a recursive enumerable set of transition rules.
  • The property representation is temporal logic (e.g. LTL, CTL).
  • The analysis algorithm answers queries in temporal logic about a program representation. We can say that a model-checking algorithm is an MFP/MOP calculation of the graph/paths generated from a program’s transition rules for answering the LTL/CTL query. Since the graph and paths might be infinite, thus answering queries is semi-decidable. We can use bounded model checking or a join operation to force it to be finite.
