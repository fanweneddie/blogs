String Analysis in JayHorn

Created2022-02-23Updated2022-02-23

This is the note of string analysis on JayHorn, which refers to paper “Towards String Support in JayHorn (Competition Contribution)” by Ali Shamakhi et al. at SV-COMP '21.

JayHorn

JayHorn is a framework for verification on Java program. As is shown in Figure 1 (copied from here, its architecture has three steps.

Figure 1: Architecture of JayHorn

  1. JayHorn leverages Soot to get Jimple IR, and simplify it by handling exception and virtual call. Note that this procedure does not alter the program behavior (more details from here).

  2. JayHorn generates constrained Horn clauses (CHC) in a modular and extensive way. For instance (copied from here), the program in Figure 2 asserts that r == 2*k in the end.

    Figure 2: the Program in an Example

    To prove this assertion, we can create a predicate for each control-location over all live variables and a horn clause for each transition. In this example, we first create one predicate p0(r,k)p_0(r, k)p0​(r,k) over variable r and k at the entry of method foo(); Then, we add a second predicate p1(r,k,i)p_1(r, k, i)p1​(r,k,i) at the entrance to the loop. So the first Horn clause is generated

    p1(0,k,o)←p0(r,k)(1)p_1(0, k, o) \leftarrow p_0(r, k) \tag{1} p1​(0,k,o)←p0​(r,k)(1)

    Likewise, we can add another two Horn clauses as loop condition check:

    p2(r,k,i)←i<k ∧ p1(r,k,i)(2)p_2(r, k, i) \leftarrow i < k \ \land \ p_1(r, k, i) \tag{2} p2​(r,k,i)←i<k ∧ p1​(r,k,i)(2)

    p3(r,k,i)←i≥k ∧ p1(r,k,i)(3)p_3(r, k, i) \leftarrow i \geq k \ \land \ p_1(r, k, i) \tag{3} p3​(r,k,i)←i≥k ∧ p1​(r,k,i)(3)

    where p2p_2p2​ is in loop body and p3p_3p3​ is after the exit of loop.

    After adding the updates in loop body at (4)(4)(4) and the assertion at (5)(5)(5), we have the final set of horn clauses.

    p1(r+2,k,i+1)←i<k ∧ p2(r,k,i)(4)p_1(r + 2, k, i + 1) \leftarrow i < k \ \land \ p_2(r, k, i) \tag{4} p1​(r+2,k,i+1)←i<k ∧ p2​(r,k,i)(4)

    r=2×k←p3(r,k,i)(5)r = 2 \times k \leftarrow p_3(r, k, i) \tag{5} r=2×k←p3​(r,k,i)(5)

  3. JayHorn supports Z3 and Eldarica as back-end Horn solvers to solve those horn clauses. The SAT of a horn formula (a conjunction of horn clauses) in propositional logic can be solved in polynomial time (as is shown in Figure 3, which is copied from here). However, the SAT of a horn formula in predicate logic is semi-decidable.

Figure 3: Polynomial Algorithm of Horn SAT Problem

String Analysis

The authors use a binary constructor cons and the constant nil to represent the recursive algebraic data type (ADT) of string. As an example, I will show how they model two string operations in horn clauses.

CompareTo

In Java, String.compareTo shows the difference of the length of strings if one of the strings is a prefix of the other, or the difference of the leftmost same-index different characters. For example, "cat".compareTo("c") = len("cat") - len("c") = 3 - 1 = 2 and "card".compareTo("cat") = 'r' - 't' = -2.

The authors use predicate P(left_string,right_string,cmp_result)P(left\_string, right\_string, cmp\_result)P(left_string,right_string,cmp_result) to recursively remove leftmost characters from both strings to reach a state where the cmp_resultcmp\_resultcmp_result is known (as is shown in Figure 4).

Figure 4: Modeling "CompareTo"

IntToStr

During the integer-to-string conversion, the pre-condition stores the rest of the input after removing the converted digits so far starting from the lowest position.

For example, for integer i=dn−1...d0‾i = \overline{d_{n-1}...d_0}i=dn−1​...d0​​, suppose so far the converted string is s=dk−1,...d0s = d_{k-1},...d_0s=dk−1​,...d0​, then the rest of the number will be r=dn−1...dk‾r = \overline{d_{n-1}...d_k}r=dn−1​...dk​​, which is stored in the pre-condition.

Result

The authors run JayHorn with different configuration on 53 problems in the SV-COMP Java track that involve strings. The results are shown in Figure 5.

Figure 5: Testing Result

As we can see,

  • RTL encoding performs better than LTR encoding.

  • the under-approximating CHC encoding of JayHorn performs better than the over-approximate encoding

  • The choice among Iterative, Recursive, or Recursive-with-precondition for string concatenation surprisingly has no effect on the results.

Limitation

The performance of this work is limited because it only covers a few string manipulation APIs in Java, given the large number of those APIs.
