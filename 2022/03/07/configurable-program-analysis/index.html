Note of Configurable Program Analysis

Created2022-03-07Updated2022-03-07

This is the note of Configurable Program Analysis (CPA), which mainly refers to “Combining Model Checking and Data-Flow Analysis”.

Relation

As I mentioned in Concepts of Program Analysis, the relation among static analysis, dataflow analysis and program verification is shown in Figure 1.

Figure 1: Relation among 3 Concepts

To be short, static analysis uses dataflow analysis and model checking, where model checking can also be used in other fields such as protocol verification.

Also, note that both data flow analysis and model checking can use abstract interpretation to converge faster.

Data Flow Analysis

The algorithm of data flow analysis computes, for a given abstract domain, a function reached that assigns to each analyzed program location an abstract data state. The algorithm is shown in Figure 1.

Figure 1: Algorithm for Dataflow Analysis

The algorithm starts by assigning the initial abstract data state to the initial program location. In every loop iteration, one program location is taken out of the waitlist and abstract successors are computed for the corresponding successor program locations. The abstract data element for the successor program location in function reached is added for the program location, or the old abstract data state is replaced by the join of the old and new abstract data states.

Model Checking

Model checking computes the nodes of an abstract reachability tree, which contains all reachable abstract states.

Figure 2: Algorithm for Model Checking

In the algorithm above, two sets reached and waitlist are initialized with a pair of the initial control-flow location and the initial abstract data state. In every iteration of the while loop, the algorithm takes one abstract state from the set waitlist and computes successors, as long as the fixed point is not reached. If there is no abstract state in the set reached that is more “abstract” than the new abstract state, then the new abstract state is added to the sets reached and waitlist.

Difference

There are two differences on algorithms of dataflow analysis and model checking. First, reached is a map in dataflow analysis, while it is a set of pairs in model checking; Second, line 8 ~ 11 in algorithms above denotes that at a joining place of program location, dataflow analysis uses join operator to over-approximate the abstract state after joining, while model checking stores all abstract states.

Intuitively, data flow analysis is designed for fast convergence but less precision, while model checking is designed for more precision but slow convergence.

Similarity

According to “Data flow analysis as model checking”, dataflow analysis is concerned with the automatic identification of program points enjoying specific properties, i.e.

programs→program points enjoying property Φprograms \rightarrow program \ points \ enjoying \ property \ \Phi programs→program points enjoying property Φ

Model checking is concerned with the automatic identification of those states of a finite state model satisfying a specific modal formula, i.e.

modal formula Φ×model→states satisfying Φmodal \ formula \ \Phi \times model\rightarrow states \ satisfying \ \Phi modal formula Φ×model→states satisfying Φ

If we identify programs with models, program points with states and program properties with modal formulas, then model checking can be seen as a special dataflow analysis.

Unification

Dirk Beyer et al. introduced configurable program analysis as a unification of dataflow analysis and model checking. A configurable program analysis D=(D,⇝,merge,stop)\mathbb{D} = (D, \rightsquigarrow, merge, stop)D=(D,⇝,merge,stop) consists of abstract domain DDD, a transfer relation ⇝\rightsquigarrow⇝, a merge operator mergemergemerge and a termination check stopstopstop. Here, merge(e,e′)merge(e,e')merge(e,e′) returns an element from e′e'e′ to ⊤\top⊤, depending on eee; stop(e,R)stop(e,R)stop(e,R) checks if abstract state eee is covered by the set of abstract states RRR. The unified algorithm is shown in Figure 7.

Figure 7: Configurable Program Analysis

Algorithm in Figure 7 puts program location into abstract states. Every iteration of the loop processes one element from the set waitlist, and computes all abstract successors for that abstract state. For every abstract successor state, the algorithm merges the new abstract state with every existing abstract state in the set reached. If the merging result is more “abstract”, then it will replace this existing abstract state. After the current abstract successor state has been merged with all existing abstract states, the stop operator determines whether the algorithm needs to store the current abstract state in the sets reached and waitlist.

In CPA, the components abstract domain, transfer relation, merge operator, and stop operator separate concern and provide a flexible way of tuning these components or exchanging them with others, which can tune the precision and efficiency.

Conclusion

Dataflow analysis and model checking are two possible methods for static program analysis. They both explore states, but handle joining point differently: dataflow analysis uses a join operator for efficiency, while model checking records different states at the same program location.

Here, the authors give a CPA algorithm to combine them together, by defining and tuning mergemergemerge operator. So that’s why they call it “configurable”.
