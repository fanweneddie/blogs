Note of Kairux

Created2022-03-22Updated2022-03-22

This is my simple note of paper “The Inflection Point Hypothesis: A Principled Debugging
Approach for Locating the Root Cause of a Failure” by Yongle Zhang et al. at SOSP 2019.

Target

The target is to identify the root cause of failures over distributed systems, which is done after reconstructing failure execution. Here, the authors define root cause as “most basic reason for an undesirable condition or problem which, if eliminated or corrected, would have prevented it from existing or occurring”. (Wait… what on earth is “most basic”??? )

Principle

The authors introduce inflection point hypothesis to solve this problem. For an instruction sequence σf\sigma_fσf​ that causes failure execution, they compare it with every failure-free execution sequence and find σp\sigma_pσp​ that shares the longest common prefix with σf\sigma_fσf​. The hypothesis is that the instruction where σf\sigma_fσf​ and σp\sigma_pσp​ deviate is the root cause of failure.

For the example in Figure 1, statement a = -1 is place that σf\sigma_fσf​ and σp\sigma_pσp​ deviate, and it is actually the root cause of this concurrent failure.

Figure 1: An Abstract Failure Example

If this hypothesis is correct, then it transforms root cause analysis into a search problem. However, since there are so many failure-free execution sequences, we cannot simply compare σf\sigma_fσf​ with all of them.

Kairux is a tool that finds that σp\sigma_pσp​ only from failure-free sequences in existing unit tests. They choose instruction sequences from unit tests, because the rich set of unit tests usually has a high coverage. Kairux does the following things:

  • removing instructions (from the sequences) that are unrelated to failure symptom,
  • using a heuristic to prioritize the unit tests to make it likely have instruction sequences in common with the failure execution,
  • splicing together subsequences from multiple threads and tests to construct σp\sigma_pσp​.

Kairux has the limitation such as

  • locating root cause only when a deviation of dataflow or control flow exists,
  • lacking bug fixing ability,
  • requiring failure to be reproducible.

Design

In order to construct σp\sigma_pσp​ without fully utilizing Σv\Sigma_vΣv​, Kairux leverages unit test suite, which has a high coverage for operations. Firstly, Kairux may require multiple tests since σf\sigma_fσf​ may involve multiple operations; Secondly, Kairux can modify test execution to make a new failure-free instruction sequence σt′\sigma_{t'}σt′​ from test suite that is closer to σf\sigma_fσf​.

The algorithm for Kairux is shown in Figure 2. It has several steps:

  1. It obtains the dynamic program slice sf‾\overline{s_f}sf​​ of failure symptom of σf\sigma_fσf​, which locates the erroneous position more precisely. This process is done by:
    1. doing a backward static analysis from inflection point, which will get a super-set of dynamic slice.
    2. setting a breakpoint at each program location in the static slice and reproducing the failure. For each breakpoint that is reached in sfs_fsf​, Kairux performs a similar dependency analysis to obtain dynamic slice.
  2. It separates sf‾\overline{s_f}sf​​ into different subsequences where each belongs to a separate thread. Hence, we can use unit tests executions to compose sps_psp​.
  3. It iteratively analyzes each thread in sf‾\overline{s_f}sf​​. In each iteration,
    1. Kairux tries to construct sp∣ts_p \mid tsp​∣t that matches the longest prefix in sf‾∣t\overline{s_f} \mid tsf​​∣t up to the instruction at ptptpt. Here, it leverages a simple sorting algorithm by ranking the number of executed functions in common with those of sf‾∣t\overline{s_f} \mid tsf​​∣t.
    2. For each highly ranked unit test, Kairux first runs it to obtain a trace sus_usu​ that only records those instructions that are in the failure’s static slice.
    3. Later, Kairux compare sf‾∣t\overline{s_f} \mid tsf​​∣t and su‾∣t\overline{s_u} \mid tsu​​∣t to find there common prefix and attempt to modify the test uuu to produce a non-failure execution with a longer common prefix to sf‾∣t\overline{s_f} \mid tsf​​∣t.
    4. After getting the non-failure instruction subsequence sp∣ts_p \mid tsp​∣t having the longest common prefix with sf‾∣t\overline{s_f} \mid tsf​​∣t, Kairux merges sp∣ts_p \mid tsp​∣t into sps_psp​ to form a longer sps_psp​. (If sps_psp​ already has instruction sequences from other threads, then Kairux merges the two instruction sequences by trying to follow the same interleaving as in the failure execution sfs_fsf​.)
    5. After analyzing thread ttt's instruction subsequence, Kairux checks to see if there is any data-flow dependency between sf‾∣t\overline{s_f} \mid tsf​​∣t and another thread t′t't′. If so, Kairux adds the tuple <sf‾∣t′,w∣t><\overline{s_f} \mid t' , w \mid t><sf​​∣t′,w∣t> to queue for next iteration.
  4. Eventually, Kairux will construct a non-failure execution sps_psp​ and its corresponding inflection point.
Figure 2: Kairux's Algorithm

Implementation

Kairux is built in Java, with Chord for static analysis and JVMTI to set breakpoints at bytecode locations.

Evaluation

The experiment checks whether Kairux can effectively locate root cause of real world failures. To do so, the authors evaluated Kairux on 10 real-world failures. The results are shown in Figure 3.

Figure 3: Evaluation Results

Overall, Kairux can successfully find root causes in 70% cases. The authors check whether failure execution σf\sigma_fσf​ and non-failure execution σc\sigma_cσc​ after bug fixing diverges at the inflection point. In 5 cases, the developers fix bug at inflection point; In 2 cases, the developers fix bugs at other points instead of inflection point; In other 3 cases, Kairux fails to locate the root cause due to lack of unit test and symptom’s out-of-code-range.

With dynamic slicing, we can effectively reduce the size from sfs_fsf​ to sf‾\overline{s_f}sf​​.

Kairux can finish the analysis in less than 32 minutes, which I believe is acceptable.

Conclusion

Based on Inflection Point Hypothesis, Kairux transforms root cause analysis into a searching problem for longest common prefix. It further leverages unit tests and program slicing as an approximation to reduce the complexity.

Although this paper tells the story in an intriguing way, the actual research workload behind is really huge. Because of its unique idea and solid implementation, this work is valuable enough to be published on SOSP. 😃
