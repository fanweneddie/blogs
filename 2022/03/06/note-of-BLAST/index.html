Note of BLAST

Created2022-03-06Updated2022-03-06

This is the note of paper “BLAST” by Dirk Beyer et al. at STTT 2007.

Abstract

BLAST (for “Berkeley Lazy Abstraction Software verification Tool”) is an automatic verification tool for checking temporal safety properties of C programs. In fact, BLAST constructs, explores, and refines abstractions of the program state space based on lazy predicate abstraction and interpolation-based predicate discovery.

Introduction

Counterexample-guided abstraction refinement (CEGAR) is a paradigm for model checking. If an counterexample of infeasible execution path is found, then it can be used to refine the model of predicates for subsequent verification attempts. CEGAR has been applied to SLAM project, and is improved in BLAST.

  • Relevant predicates are discovered locally and independently at each program location as interpolants between the past and the future fragments of an infeasible error path.
  • Instead of constructing an abstraction of the program which tracks all relevant predicates, the discovered new predicates are added and tracked locally in some parts of a tree that represents the abstract executions of the program, namely, in those parts where the infeasible error path occurred.

When in the absence of correctness assertions, coverage goal is required to find the set of all program locations qqq that program execution can reach with the value of a predicate ppp being true. Moreover, test vector, a map from program inputs to values that force the program execution along the given path, is also useful.

For BLAST, it can find the reachable program locations when ppp is true; And if location qqq is reachable where predicate ppp is true, then we can automatically produce a test vector that witnesses the truth of ppp at qqq.

Idea

ART

To prove that the error configuration is never reached, Blast constructs an abstract reachability tree (ART). Each node n:(q,s,φ)n: (q, s, \varphi)n:(q,s,φ) of the ART is labeled with a CFA location qqq, the current call stack sss and a reachable region φ\varphiφ representing a set of reachable states.

We use post(φ,op)post(\varphi, op)post(φ,op) to be the region reachable from φ\varphiφ by executing operation opopop. An ART is complete if

  • the root is labeled with the initial states of the program;
  • the tree is closed under postconditions, i.e. for every internal node n:(q,s,φ)n:(q,s,\varphi)n:(q,s,φ) of the tree with φ\varphiφ is satisfiable.
  • for every leaf node n:(q,s,φ)n:(q,s,\varphi)n:(q,s,φ), either qqq has no outgoing edge in its CFA, or φ\varphiφ is not satisfiable, or there exists an internal tree node n′:(q,s,φ′)n':(q,s,\varphi')n′:(q,s,φ′) s.t. φ⇒φ′\varphi \Rightarrow \varphi'φ⇒φ′.

A complete ART over-approximates the set of reachable states of a program. A complete ART is safe w.r.t. a configuration (p,q)(p,q)(p,q), where qqq is a CFA location and qqq is an error region, if for every node n:(q,⋅,φ)n:(q,\cdot, \varphi)n:(q,⋅,φ), we have φ∧p\varphi \land pφ∧p is unsatisfiable.

CEGAR

Blast starts with a coarse abstraction of the state space and attempts to construct a complete ART with the coarse abstraction. If this complete ART is safe for the error configuration, then the program is safe. Else, Blast runs a counterexample-analysis algorithm that determines whether the path to the error configuration is feasible.

The refinement procedure refines the abstraction locally, and the search is resumed on the nodes of the ART where the abstraction has been refined.

ART Construction

The ART construction proceeds by unrolling the CFAs and keeping track of the reachable region at each CFA location. At first, we start with the initial location of the program with the reachable region true. In the first iteration, since we do not track any facts about variable values, all reachable regions are over-approximated by true (so BLAST finds that the error location may be reachable in the example).

Then, BLAST invokes the counterexample-analysis algorithm to check if the error path is feasible in the concrete program, or whether it arises because the current abstraction is too coarse. For the abstract error path, BLAST creates a set of constraints called Path Formula, which is satisfiable iff the path is feasible in the concrete program.

If the path formula is unsatisfiable, then BLAST uses predicate-discovery algorithm to find new predicates that must be added to the abstraction in order to rule out the infeasible error path.

Limitation

There are some technique reasons why BLAST may give false alarm or unsound results.

  • The decision procedures of BLAST implements uninterpreted functions.
  • BLAST assumes that the memory is laid out as a logical array, and it further assumes that all pointer arithmetics is safe. Also, BLAST uses a flow-insensitive may-alias analysis to deal with pointer updates.
  • BLAST treats library calls by assuming that the return values can be arbitrary.
  • The predicates tracked by BLAST do not contain logical quantifiers.
