Concepts of Program Analysis

Created2022-03-05Updated2022-03-05

This is my understanding of several concepts of program analysis.

Program Analysis

Program analysis is the process of automatically analyzing the behavior of computer programs regarding a property such as correctness, robustness, safety and liveness. It can be performed during runtime as dynamic program analysis, without executing the program as static program analysis, or in a combination of both (from wiki).

Dynamic Program Analysis

Dynamic program analysis should be performed by executing programs on a real or virtual processor (from wiki). For instance, testing, monitoring and program slicing belong to dynamic program analysis.

Here, we mainly talk about static program analysis.

Static Program Analysis

Static program analysis aims to automatically answer questions about the possible behaviors of programs without executing any programs (from spa). We can use some implementation techniques such as data-flow analysis and formal verification. However, according to Rice’s theorem, there is no mechanical method that can always answer truthfully whether an arbitrary program may or may not exhibit runtime errors (from wiki), otherwise the halting problem is decidable. Therefore, we may use abstract interpretation as an approximation to make this analysis scalable.

Abstract Interpretation

Abstract interpretation is a theory of sound approximation of the semantics of computer programs (from wiki). The domain space of concrete analysis is sometimes too large to scale. Therefore, we can do analysis in an abstract domain as an over-approximation to reduce this cost. In order to make this abstract analysis sound, we need to define this abstract domain and the mapping function carefully.

Take the sign analysis as an example. Instead of using concrete values to represent a variable, we can use the sign to represent it. Data-flow Analysis

Data-flow analysis refers to a body of techniques (such as constant propagation) that derive information about the flow of data along program execution path (from dragon book). A simple way to perform data-flow analysis is to set up data-flow equations for each node of control flow graph and do iteration until a fixpoint is reached(from wiki).

By using abstract interpretation, each variable at each program location is represented by an element in a semi-lattice. Since the state of a variable at a program location moves monotonously, then the fixpoint algorithm must converge in O(nvh)O(nvh)O(nvh) time where nnn is the number of program locations, vvv is the number of variables and hhh is the height of semi-lattice (if finite).

Let’s take constant propagation as an example. Constant propagation checks whether each variable at each program location is a constant value, i.e. the variable has the same value no matter what inputs are provided and no matter what code path was followed (from mozilla wiki). Its semi-lattice is shown in Figure 2-1, where the top element represents “undefined value” and bottom element represents “not a constant” (from dragon book).

Figure 2-1: Lattice for Constant Propagation

For a variable xxx at a program location lll, its abstract value depends on the statement sss at lll:

  • If sss is not an assignment statement, then the value of xxx does not change.
  • If sss is a constant assignment statement x=cx = cx=c, then the value becomes the corresponding m(c)m(c)m(c) in lattice.
  • If sss is an assignment statement x=y+zx = y + zx=y+z, then for each possible input value of yyy, the value of xxx does not get bigger as zzz gets smaller (as is shown in Figure 2-2). By induction, we know that the prime value of xxx cannot be bigger than the value of xxx.

Hence, constant propagation is monotone and it must converge.

Formal Verification

Formal verification is the process of checking whether a design satisfies some specifications or properties, by using mathematical techniques such as model checking and deductive verification (from ptolemy). Compared with simulation and testing, formal verification conducts an exhaustive exploration of all possible behaviors (from Clarke).

Model checking is a method for automatically checking whether a finite-state model of a system meets a given specification (from wiki) and it consists of a systematically exhaustive exploration of the mathematical model. Here, the specification can be described in Temporal Logics and the model includes finite-state automata.

Note that formal verification can be used in program verification, protocol verification and computer hardware verification.

Soundness and Completeness

In mathematical logic, soundness means that each formula that can be proved is logically valid w.r.t. the semantics of the system, i.e. syntactic entailment implies semantic entailment (Γ⊢p⇒Γ⊨p\Gamma \vdash p \Rightarrow \Gamma \vDash pΓ⊢p⇒Γ⊨p) (from wiki).

On the other hand, completeness means that every formula having a property can be derived from the system, i.e. semantic entailment implies syntactic entailment (Γ⊨p⇒Γ⊢p\Gamma \vDash p \Rightarrow \Gamma \vdash pΓ⊨p⇒Γ⊢p) (from wiki).

For algorithms, we say that an algorithm is sound, if this algorithm does not yield any results that are untrue; On the other hand, we say that an algorithm is complete, if this algorithm guarantees to return a correct answer for any arbitrary input. Briefly speaking, soundness says that if an answer is returned, then it is true; Completeness says that if an answer is true, then it is returned (from stackexchange).

Take the solver for simple equation x2=1x^2 = 1x2=1 as an example. It is obvious that the completely correct solution is {1,−1}\{ 1, -1 \}{1,−1}. If the solver returns solution {1}\{ 1 \}{1}, then this solver is sound (since it does not get a wrong answer) but not complete (since it omit another correct answer); If the solver returns solution {−1,0,1}\{ -1, 0, 1 \}{−1,0,1}, then this solver is complete (since all correct answers are returned) but not sound (since it returns an incorrect answer); If the solver returns {1,−1}\{ 1, -1 \}{1,−1}, then it is sound and complete.

Analyses are expected to be sound in that their result models all possible executions of the program under analysis, but they actually will also model behaviors that do not actually occur in any program execution. This unsoundness is caused by the compromise to make analysis scalable to large programs with complex language features. In 2015, experts used soundiness to capture the balance that the analysis is mostly sound, with well-identified unsound treatments to hard/specific language features (from soundiness).
