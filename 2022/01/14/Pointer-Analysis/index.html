Pointer Analysis

Created2022-01-14Updated2022-01-14

Overview

I refer to Alias Analysis for Object-Oriented Programs for this note.

Pointer Analysis is also called Alias Analysis.

We need to check whether two variable expressions point to/reference the same memory location.

A points-to analysis computes an over-approximation of the heap locations that each program pointer may point to.

We have the version of context-insensitive and context-sensitive.

Context-insensitive

We have two flow-insensitive and context-insensitive algorithms: Andersen and Steensgaard.

Since they are flow-insensitive, we can execute those algorithms on AST.

Andersen

Source: Pointer Analysis in cs252 in Havard

Introduction

Andersen algorithm is a flow-insensitive and context-insensitive algorithm. It is a constraint-based algorithm and focuses on subset constraints of assignment:

  • a = &b: loc(b)∈pts(a)loc(b) \in pts(a)loc(b)∈pts(a)
  • a = b: pts(b)⊆pts(a)pts(b) \subseteq pts(a)pts(b)⊆pts(a)
  • a = *b: ∀v∈pts(b),pts(v)⊆pts(a)\forall v \in pts(b), pts(v) \subseteq pts(a)∀v∈pts(b),pts(v)⊆pts(a)
  • *a = b: ∀v∈pts(a),pts(b)⊆pts(v)\forall v \in pts(a), pts(b) \subseteq pts(v)∀v∈pts(a),pts(b)⊆pts(v)

Implementation

We can also use a workList graph algorithm to solve this problem.

The initialization of graph(based on assignment) is like this:

  • a = &b, we have {b}⊆a\{b\} \subseteq a{b}⊆a.
  • a = b, we have b⊆ab \subseteq ab⊆a and we build an edge b→ab \rightarrow ab→a.
  • a = *b, we have ∗b⊆a^*b \subseteq a∗b⊆a.
  • *a = b, we have b⊆ ∗ab \subseteq \ ^*ab⊆ ∗a.

It is flow-insensitive because it is an iteration algorithm and does not care about the sequence of flow.

The complexity of Andersen algorithm is O(n3)O(n^3)O(n3), where nnn is the number of nodes in graph.

We can reduce nnn by collapsing SCC to single node in point-to graph.

Introduction

Context-sensitive

Context-sensitive points-to analysis analyzes a method mmm for each calling context that arises at call sites of mmm.

A calling context is some abstraction of the program states at may arise at a call site.

We assume that a method mmm has formal parameters mthism_{this}mthis​ for the receiver and mp1,...,mpnm_{p_1},...,m_{p_n}mp1​​,...,mpn​​ for the parameters, and a variable mretm_{ret}mret​ to hold the return value.

Call Strings

A standard technique to distinguish contexts is via call strings.

Call strings are typically represented as a sequence of call site identifiers, corresponding to a (partial) call stack.

In this way, we can set function selector(m′,c,j,argvals)selector(m',c,j,argvals)selector(m′,c,j,argvals) as

selector(_,[j0,j1,...,jn],j,_)={[j,j0,j1,...,jn]}selector(\_,[j_0,j_1,...,j_n],j,\_) = \{ [j,j_0,j_1,...,j_n] \} selector(_,[j0​,j1​,...,jn​],j,_)={[j,j0​,j1​,...,jn​]}

Object Sensitivity

It uses the (abstract) objects passed as the receiver argument to the method.

By using receiver objects to distinguish contexts, an object-sensitive analysis can avoid conflation of operations performed on distinct objects.

Therefore, context is a list of allocation sites of receiver objects.

Now, we can define selectorselectorselector function as
