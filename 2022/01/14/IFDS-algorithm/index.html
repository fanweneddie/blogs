IFDS Algorithm

Created2022-01-14Updated2022-01-15

This is the note of IFDS algorithm by Thomas Reps, Susan Horwitz and Mooly Sagiv at popl95.

Introduction

Intraprocedural: “precise” means “meet-over-all-paths”

Interprocedural: “precise” means “meet-over-all-valid-paths”

The paper provides a polynomial-time algorithm for finding precise solutions to a general class of interprocedural dataflow analysis problems. In this problem, the set of dataflow facts D is a finite set and the dataflow functions distribute over the meet operator(union or intersection)

So we call it interprocedural, finite, distribute, subset (IFDS) problems.

IFDS

Graph

A program is represented using a directed graph G∗=(N∗,E∗)G^* = (N^*,E^*)G∗=(N∗,E∗) , where G∗G^*G∗ is called a super graph.

G∗G^*G∗ consists of a collection of flow graphs G1G_1G1​, G2G_2G2​,…(one for each procedural) and GmainG_{main}Gmain​ (for the main procedural of the program)

Each flowgraphGp has a unique start node sps_psp​ and a unique exit node epe_pep​.

The procedural call is represented by two nodes: a call node ccc and a return-site node rrr.

In each GiG_iGi​, there are ordinary intraprocedural edges that connects the nodes of the individual flowgraphs.

G∗G^*G∗ has three edges,

  • An intraprocedural call-to-return-site edge from ccc to rrr.
  • An interprocedural call-to-start edge from ccc to the start node of the called procedural
  • An interprocedural exit-to-return-site edge from the exit node of the called procedural to rrr.

For call-to-return-site and exit-to-return-site edges, they permit the information about local variables that holds at the call site to be combined with the information about the global variables that holds at the end of the called procedural.

Conversion

How to convert IFDS to realizable-path graph reachability problems?

For each instance IPIPIP in IFDS problem, we construct a graph GIP#G_{IP}^\#GIP#​ and an instance of realizable-path graph reachability problem. The edges of IPIPIP corresponds to the representation relations of the dataflow functions on the edges of G∗G^*G∗.

fact

Dataflow-fact ddd holds at supergraph node nnn iff there is a “realizable path” from a distinguished node in GIP#G_{IP}^\#GIP#​ to the node in GIP#G_{IP}^\#GIP#​ that represents the fact ddd at node nnn.

Tabulation Algorithm

Tabulation Algorithm is an efficient algorithm for Realizable-Path Reachability Problem, which is based on dynamic programming. This function is not path-sensitive because it does not iterate over every possible execution path???

Four functions

  • returnSite: call node →\rightarrow→ return site node
  • procOf: node →\rightarrow→ the name of its enclosing procedure
  • calledProc: call node →\rightarrow→ the name of called procedure
  • caller: procedure name →\rightarrow→ the set of call nodes that call to that procedure

Edges

  • PathEdge: a set to record the existence of path edges, which represents the suffix of the same-level realizable paths in graph GIP#G_{IP}^\#GIP#​.
  • SummaryEdge: a set to record the existence of summary edges, which represent same-level realizable paths that run from nodes of the form <n,d1><n,d_1><n,d1​> to <returnSite(n),d2><returnSite(n), d_2><returnSite(n),d2​>, where n∈Calln \in Calln∈Call. So summary edges represent the partial information about how the dataflow value after a call depends on the dataflow value before the call.

The Tabulation Algorithm is a worklist algorithm that accumulates sets of PathEdge and SummaryEdge. At last, we can check the path edge and get the elements in MVP
