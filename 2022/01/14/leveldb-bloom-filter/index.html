Leveldb Bloom Filter

Created2022-01-14Updated2022-01-15

This is the simple analysis of bloom filter in leveldb.

Basic Concepts

A bloom filter checks whether a key

  • must not in a set, or
  • possibly in a set

And it consists of

  • nnn keys
  • a space vvv of mmm bits, all initialized as 0
  • kkk independent hash functions: h1,h2,...,hkh_1, h_2, ..., h_kh1​,h2​,...,hk​, and the result of hash is {0,1,...,m−1}\{ 0,1,...,m-1 \}{0,1,...,m−1}

When building a set, we do the following steps: for a key keykeykey, we compute its hash result: h1(key),h2(key),...,hk(key)h_1(key), h_2(key),...,h_k(key)h1​(key),h2​(key),...,hk​(key), and set those corresponding bits to be 1. For example, if k=3k = 3k=3 and h1(key)=0,h2(key)=5,h3(key)=10h_1(key) = 0, h_2(key) = 5, h_3(key) = 10h1​(key)=0,h2​(key)=5,h3​(key)=10, then v[0]=v[5]=v[10]=1v[0] = v[5] = v[10] = 1v[0]=v[5]=v[10]=1.

When we are checking a key, we do the following steps: for a key keykeykey, we still compute its hash result: h1(key),h2(key),...,hk(key)h_1(key), h_2(key),...,h_k(key)h1​(key),h2​(key),...,hk​(key), and check whether those corresponding bits are all 1. If so, keykeykey may be in the set; or, keykeykey must not in the set.

As the number of keys grows, there are more bits being set as 1. For a key that is not in a set, all of its corresponding bits may be set by other keys, so a false-positive case may happen.

Optimal number of hash function

The bigger mmm and kkk is, the smaller nnn is, the lower false positive rate it has. This is kinda intuitive:

  • when mmm is bigger, we have more bits to show whether a key is in, so the chance that these bits are all set by other keys is lower;
  • when kkk is bigger, we have more “criteria” to represent a key, so it is less likely that this key is forged by other keys;
  • when nnn is smaller, there are fewer keys and the possibility of hash collision is lower (obvious).

And we have a claim that: the optimal number of hash function is k=mnln⁡2k = \frac{m}{n} \ln{2}k=nm​ln2.

pf:

For a hash function hih_ihi​ and a given bit in vvv, the possibility of setting it as 1 is (suppose it is universe)

Pi(1)=1mP_i(1) = \frac{1}{m} Pi​(1)=m1​

and the possibility of not setting it is

Pi(0)=1−1mP_i(0) = 1 - \frac{1}{m} Pi​(0)=1−m1​

Since there are kkk independent hash functions and nnn keys, the possibility of not setting this bit is

P(0)=(1−1m)knP(0) = (1 - \frac{1}{m}) ^{kn} P(0)=(1−m1​)kn

Given the lemma:

limx→∞(1−1x)−x=elim_{x \rightarrow \infty} (1 - \frac{1}{x})^{-x} = e limx→∞​(1−x1​)−x=e

Since mmm is usually big, We have

P(0)≈e−knmP(0) \approx e^{\frac{-kn}{m}} P(0)≈em−kn​

So

P(1)≈1−e−knmP(1) \approx 1 - e^{\frac{-kn}{m}} P(1)≈1−em−kn​

Now we talk about the false positive case. For a key, it is not in a set, but its corresponding kkk bits are all set as 1. So the possibility of false-positive is

Pfp=P(1)k≈(1−e−knm)kP_{fp} = {P(1)}^k \approx (1 - e^{\frac{-kn}{m}})^k Pfp​=P(1)k≈(1−em−kn​)k

For simplicity, we set x=e−knm∈(0,1)x = e^{\frac{-kn}{m}} \in (0,1)x=em−kn​∈(0,1), so our goal is to minimize

Pfp=(1−x)k=(eln⁡(1−x))k=ekln⁡(1−x)P_{fp} = (1-x)^k = (e^{\ln(1-x)})^k = e^{k \ln(1-x)} Pfp​=(1−x)k=(eln(1−x))k=ekln(1−x)

which means that we should minimize

f(x)=kln⁡(1−x)f(x) = k \ln(1-x) f(x)=kln(1−x)

Note that x=e−knmx = e^{\frac{-kn}{m}}x=em−kn​, so k=−mnln⁡xk = - \frac{m}{n} \ln{x}k=−nm​lnx and

f(x)=−mnln⁡xln⁡(1−x)f(x) = - \frac{m}{n} \ln{x} \ln(1-x) f(x)=−nm​lnxln(1−x)

By symmetry, when x=12x = \frac{1}{2}x=21​, f(x)f(x)f(x) gets its minimum value. So now we have

k=−mnln⁡12=mnln⁡2k = - \frac{m}{n} \ln{\frac{1}{2}} = \frac{m}{n} \ln2 k=−nm​ln21​=nm​ln2

Implementation in leveldb

For leveldb, bloom filter is divided into policy and implementation. Policy is defined in class FilterPolicy in leveldb/filter_policy.h, and implementation is defined in class FilterBlockBuilder and FilterBlockReader in table/filter_block.h.

Policy

class FilterPolicy is an “abstract” father class of the policy. Usually, we use BloomFilterPolicy in util/bloom.cc, which extends FilterPolicy.

BloomFilterPolicy contains 2 fields:

  • bits_per_key_: As mn\frac{m}{n}nm​ where mmm is the number of bits and nnn is the number of keys.
  • k_: the number of hash functions.

And it has 3 methods:

  • constructor: given bits_per_key_, it does initialization.

    According to the theorem above, the optimal number of hash functions is k=mnln⁡2≈0.69mn=0.69×k = \frac{m}{n} \ln{2} \approx 0.69 \frac{m}{n} = 0.69 \timesk=nm​ln2≈0.69nm​=0.69×bits_per_key_. So line 3 just uses this conclusion to initialize k_. Note that line 4~5 adjusts k_ to make sure it is in range [1,30][1,30][1,30], because if k_ is too large, the cost of computing hash value may be too high.

  • CreateFilter(): given n keys, build their bloom filter into a destination segment.

    Here, keys is an array that stores n keys and dst is a segment that stores the result of bloom filter.

    • Line 3 ~ 10 computes the total number of bits and bytes (in bloom filter) for n keys in keys, which are stored in bits and bytes.
    • Line 12 ~ 15 enlarges capacity and finds the place to append new bits. Note that dst has stored many bloom filters, so we should get the offset to this bloom filter (to be built). Line 14 stores k_ at the end of dst; After line 15, the place is recorded as array.
    • Line 16 ~ 26 sets bits in array. For each key, it uses a double-hashing method, and rotates bits for k_ times to set bits.

    At last, the calculated bloom filter is appended in dst.

  • KeyMayMatch(): given a key and a segment(as bloom filter), check whether the key may be in or must not be in.

    • line 2 ~ 15 gets the basic information of this bloom filter. For example, its length (in len), the number of hash functions (in k).
    • line 17 ~ 24 computes the hash value, which is similar to that in method CreateFilter(). When any of the bit is 0, it will return false.

    Note that hash function is the same.

Implementation

We use class FilterBlockBuilder to build filter blocks, and FilterBlockReader to read blocks.

FilterBlockBuilder

FilterBlockBuilder has these fields:

  • const FilterPolicy* policy_: the policy(defined above), which is initialized by the policy in options.
  • std::string keys_: the vector of serialized keys.
  • std::vector<size_t> start_: the vector of offset of each key.
  • std::string result_: the vector of serialized bloom filter as result.
  • std::vector<Slice> tmp_keys: storing temporary keys when building bloom filters.
  • std::vector<uint32_t> filter_offsets_: the offsets of each bloom filters.

And it has these methods:

  • constructor: it only initializes policy_.

  • AddKey(): add a key into this object. This method is invoked after we have added this key into a data block.

    • line 2 ~ 4 updates keys_ and start_ of this object.

  • StartBlock(): start to build a filter block. This method is invoked when we are flushing a data block into disk.

    • line 2 gets the number of filters (that is what filter_index means). Here, block_offset is the offset of filter block. Note that the format of a sstable is

      Since one sstable only has one filter block (meta block), so the offset of the filter block denotes the total size of data block.

    • kFilterBase =1≪11=210=2K= 1 \ll 11 = 2^{10} = 2K=1≪11=210=2K, which means that we build a filter segment when the size of new data reaches 2 KB (If the size of new data is less than 2 KB, it would not build a filter block for this data). Here, “new” means the data (in data block) which has not generated its bloom filter. The purpose of this approach is to limit the number of bloom filters.

    • line 3 ~ 6 generates filters for each segment of keys. Note that line 2 does sanity check, because the number of all keys must be bigger than the number of keys that have built their filters.

  • GenerateFilter(): generate a filter

    • line 2 gets the number of keys.
    • line 3 ~ 7 just lets this segment empty when there is no key.
    • line 10 ~ 16 record the size of current key segment into start_, and use this info to copy keys from keys_ to tmp_keys.
    • line 19 ~ 20 record the size of current bloom filter into filter_offsets_, and leverages policy_ to create a filter and append it into result_.
    • line 22 ~24 clears the data structures since their key segment can no longer be used.

  • Finish(): finish generating filter block. This method is invoked when the construction of a sstable is finished.

    • line 6 ~ 10 appends the offset of each filter segment into result_
    • line 12 ~ 13 appends the size of bloom filter and the kFilterBaseLg into result_

FilterBlockReader

FilterBlockReader has these fields:

  • const FilterPolicy* policy_: the bloom filter policy
  • const char* data_: Pointer to filter data (at block-start)
  • const char* offset_: Pointer to beginning of offset array (at block-end)
  • size_t num_: Number of entries in offset array, which denotes the number of bloom filter segments.
  • size_t base_lg_: Encoding parameter (see kFilterBaseLg in this file)

And it has these methods:

  • Constructor: Given the policy and a buffer of contents for initialization

    • According to the format of bloom filter, line 3 ~ 11 gets the parameters of this bloom filter.

  • KeyMayMatch(): check whether the given key key at offset block_offset may be in the block. Note that block_offset denotes the offset of data block that key may be in, which can help get the exact bloom filter segment.

    • line 2 gets the ID of the bloom filter segment (record it in index).
    • line 4 ~ 8 finds the exact bloom filter segment and check the filter.
    • line 9 ~ 12 denotes the case that the corresponding bloom filter segment is empty, so the key must not be in the block.

Example

Suppose we will write 4 data blocks, and their sizes are 1 KB, 3 KB, 2 KB, 5 KB, respectively (Here, the threshold size of data block is not necessarily 4 KB). The initial state is shown in Figure 1: Rectangles represent each data block, and keys_ stores the temporarily added keys, and offset_ represents the offsets of each bloom filter segment, and result_ represents the bloom filter.

Figure 1: Initial state

There are several steps to build the bloom filter:

  1. Data block 1 is ready to be flushed (As is shown in Figure 2).

    Figure 2: Data block 1 to be flushed

    Keys from data block 1 have been stored in keys_. Now the total number of filter segment is ⌊1 KB2 KB⌋=0\lfloor \frac{1 \ KB}{2 \ KB} \rfloor = 0⌊2 KB1 KB​⌋=0, and current number of filter segment is 0. So we don’t create a new filter segment, and the result is the same as Figure 2.

  2. Data block 2 is ready to be flushed (As is shown in Figure 3).

    Figure 3: Data block 2 to be flushed

    Keys from data block 1~2 have been stored in keys_. Now the total number of filter segment is ⌊1 KB + 3 KB2 KB⌋=2\lfloor \frac{1 \ KB \ + \ 3 \ KB}{2 \ KB} \rfloor = 2⌊2 KB1 KB + 3 KB​⌋=2, and current number of filter segment is 0. So we need to create 2 new filter segments, where the first segment stores the bloom filter of keys_, and the second filter is just empty. The result is shown in Figure 4.

    Figure 4: Data block 2 has been flushed

    In Figure 4, keys_ has been used to generate a bloom filter segment in result_. Since keys_ will be no longer used, it is cleared. Since there should be two segments, we append two elements in offset_: offset_[0] stores the offset to the first filter, and offset_[1] stores the offset to the second filter (Though not built yet). In other words, offset_[0] points to the newly added filter (As is shown by the red arrow), and offset_[1] helps nothing (As is shown by the blue arrow).

  3. Data block 3 is ready to be flushed (As is shown in Figure 5).

    Figure 5: Data block 3 to be flushed

    Similar to step 2, keys from data block 3 have been stored in keys_. Now the total number of filter segment is ⌊1 KB + 3 KB + 2KB2 KB⌋=3\lfloor \frac{1 \ KB \ + \ 3 \ KB \ + \ 2 KB}{2 \ KB} \rfloor = 3⌊2 KB1 KB + 3 KB + 2KB​⌋=3, and current number of filter segment is 2. So we need to create 1 new filter segment. The result is shown in Figure 6.

    Figure 6: Data block 3 has been flushed

    Here, offset_[2] points to the newly added filter.

  4. Data block 4 is ready to be flushed (As is shown in Figure 7).

    Figure 7: Data block 4 to be flushed

    Similar to step 2, keys from data block 4 have been stored in keys_. Now the total number of filter segment is ⌊1 KB + 3 KB + 2KB + 5KB2 KB⌋=5\lfloor \frac{1 \ KB \ + \ 3 \ KB \ + \ 2 KB \ + \ 5 KB}{2 \ KB} \rfloor = 5⌊2 KB1 KB + 3 KB + 2KB + 5KB​⌋=5, and current number of filter segment is 3. So we need to create 2 new filter segments. The result is shown in Figure 8.

    Figure 8: Data block 4 has been flushed

    Here, offset_[3] points to the newly added filter, and offset_[4] helps nothing.

  5. All data blocks have been built, so we just append offset_ into result_ and end up building a serialized bloom filter (As is shown in Figure 9).

Figure 9: Bloom filter has been built

Now the bloom filter has been built. Let’s see the procedure of searching a key.

For a key, after simply searched in index block, if

  • its index is 0 (the offset of data block 1, which means that the key may be in data block 1), so the filter segment is ⌊02 KB⌋=0\lfloor \frac{0}{2 \ KB} \rfloor = 0⌊2 KB0​⌋=0. Then, we can leverage offset_[0] to access filter 1 and invoke method KeyMayMatch() to search the key.
  • its index is 1 KB (the offset of data block 2, which means that the key may be in data block 2), so the filter segment is ⌊1 KB2 KB⌋=0\lfloor \frac{1 \ KB}{2 \ KB} \rfloor = 0⌊2 KB1 KB​⌋=0. Then, we can leverage offset_[0] to access filter 1 and invoke method KeyMayMatch() to search the key.
  • its index is 4 KB (the offset of data block 3, which means that the key may be in data block 3), so the filter segment is ⌊4 KB2 KB⌋=2\lfloor \frac{4 \ KB}{2 \ KB} \rfloor = 2⌊2 KB4 KB​⌋=2. Then, we can leverage offset_[2] to access filter 2 and invoke method KeyMayMatch() to search the key.
  • its index is 6 KB (the offset of data block 4, which means that the key may be in data block 4), so the filter segment is ⌊6 KB2 KB⌋=3\lfloor \frac{6 \ KB}{2 \ KB} \rfloor = 3⌊2 KB6 KB​⌋=3. Then, we can leverage offset_[3] to access filter 3 and invoke method KeyMayMatch() to search the key.
