Interprocedural Analysis

Created2022-01-14Updated2022-01-14

This article is the summary of chapter 8 of “static program analysis” by Anders Moller and Michael I. Schwartzbach.

Interprocedural CFG

For a procedural call X = g(a1,...,an), in procedural f we have

  • call node of g
  • entry node of g
  • exit node of g
  • after-call node of g

Note that the connection between the call node and the after-call node is represented by a special edge(not succsuccsucc or predpredpred).

Context Insensitive

We take Sign Analysis as an example.

In intraprocedural Sign Analysis, the lattice is a map lattice: States=Vars→SignStates = Vars \rightarrow SignStates=Vars→Sign.

For interprocedural Sign Analysis, we can make some change.

Context Sensitive

Interprocedurally invalid paths: dataflow from one call node propagates through the function body and returns not only at the matching after-call node.

A naive approach is to use function-cloning, but it will increase the program size significantly. Actually, we can instead encode the relevant information to distinguish the different calls by the use of more expressive lattices.

Context-insensitive dataflow analysis can be expressed as a lattice StatesnStates^nStatesn where States=Var→SignStates = Var \rightarrow SignStates=Var→Sign and n=∣Nodes∣n = |Nodes|n=∣Nodes∣.

However, context-sensitive dataflow analysis uses a lattice of the form

(Contexts→lift(States))n(Contexts \rightarrow lift(States))^n (Contexts→lift(States))n

where ContextsContextsContexts is a set of call contexts.

Since some call contexts are not feasible, they can only map to the bottom element of lift(States)lift(States)lift(States), which is denoted as unreachableunreachableunreachable.

Note that if ContextsContextsContexts is a singleton set, then it becomes a context-insensitive analysis. If we set Contexts=StatesContexts = StatesContexts=States, then it allows full context-sensitivity.

Take assignment statement X = E as an example,

  • In intraprocedural analysis, we have

    [ ⁣[v] ⁣]=JOIN(v)[X→eval(JOIN(v),E)][\![v]\!] = JOIN(v)[X \rightarrow eval(JOIN(v),E)] [[v]]=JOIN(v)[X→eval(JOIN(v),E)]

  • In interprocedural analysis, we set the calling context of the method that X = E is in as ccc.

Here we introduce two context sensitive approaches: Call Strings and Functional Approach.

Call Strings

Let CallsCallsCalls be the set of call nodes in the CFG, then we can define contextscontextscontexts

Contexts=Calls≤kContexts = Calls^{\leq k} Contexts=Calls≤k

where kkk is a pre-defined positive integer, meaning the maximum length of call chain.

Here, a tuple (c1,c2,...,cm)∈Calls≤k(c_1,c_2,...,c_m) \in Calls^{\leq k}(c1​,c2​,...,cm​)∈Calls≤k identifies the topmost mmm call sites on the call stack.

Since kkk is finite, mmm as the length of tuple is also finite, which ensures that ContextsContextsContexts is also finite.

Functional Approach

As we can see, the Call String method analyzes the method lll times where lll is the number of call sites of that method.

Rather than distinguishing calls based on control flow information from the call stack, we can use Functional Approach to distinguish calls based on data from the abstract states at the calls. That is, we can set

Contexts=StatesContexts = States Contexts=States

And the lattice becomes

(States→lift(States))n(States \rightarrow lift(States))^n (States→lift(States))n

For each CFG node vvv, its lattice element is a map m:States→lift(States)m: States \rightarrow lift(States)m:States→lift(States). If the current function containing vvv is entered in the state that matches sss, then m(s)m(s)m(s) is the possible states at vvv. If there is no execution of the program where the function is entered in a state that matches sss and reaches vvv, then m(s)=unreachablem(s) = unreachablem(s)=unreachable at vvv.
