Note of Upgrade Failures

Created2022-01-15Updated2022-01-15

This is the note of paper “Understanding and Detecting Software Upgrade Failures in Distributed Systems” by Yongle Zhang et al. at SOSP’21.

Introduction

Distributed system software upgrade is one of the most disruptive maintenance tasks. Software upgrade failures is problematic in:

  • Large scale: Since upgrade is typically performed on the entire system, so an upgrade failure can often paralyze the entire cluster.
  • Vulnerable context: Failures under the context of no-service and partial service are particularly difficult to mask.
  • Persistent impact: an upgrade failure can corrupt system state persistently that cannot be easily recovered by rolling back to the old version.
  • Difficult to expose: hard to use traditional testing.

This paper provides the first in-depth analysis on software upgrade failures based on 123 real-world upgrade failures from widely-used distributed systems. For each case, we carefully analyzed its report and source code to thoroughly understand the symptom severity, the root cause, and the propagation chain in between.

Case studies

Severity of upgrade failures

This part finds that upgrade failures can cause big problems with huge cost.

Finding 1:

Upgrade failures have significantly higher priority than regular failures.

Finding 2:

The majority (67%) of upgrade failures are catastrophic.

Finding 3:

Most (70%) upgrade failures have easy-to-observe symptoms like node crashes or fatal exceptions.

Finding 4:

The majority (63%) of upgrade bugs were not caught before code release.

Root causes of upgrade failures

Finding 5:

About two thirds of upgrade failures are caused by interaction between two software versions that hold incompatible data syntax or semantics assumption.

To be specific, close to 2/3 of incompatibility is about syntax difference of the same data in different versions, and the remaining 1/4 is about semantic difference.

Also, 60% of incompatibility is centered on persistent storage data, while 40% is centered on network messages.

Finding 6:

Close to 20% of syntax incompatibilities are about data syntax defined by serialization libraries or enum data types. Given their clear syntax definition interface, automated incompatibility detection is feasible.

For serialization library data, One common problem is that the new version adds a required data member for a class. Then, during a rolling upgrade, the serialization library in a new node cannot find this data member in the data generated by an old node and triggers an upgrade failure.

For enum data, the problem is caused by wrong index after some insertion in new version.

Finding 7:

Most (about 80%) data syntax incompatibilities are caused by missing or incomplete deserialization functions for system-specific data.

Finding 8:

Close to two thirds of of data semantics incompatibilities are caused by incomplete version checking and handling.

Actually, each version of source code should have their own distinct version ID, and each version ID should be mapped to a set of features that are supported in the parsing logic.

Triggers of the failure

Finding 9:

All but 14 upgrade failures can be triggered by consecutive major or minor versions.

Therefore, we just need to test O(N)O(N)O(N) combinations of consecutive versions to expose a vast majority of the upgrade failures.

Finding 10:

All of the upgrade failures require no more than 3 nodes to trigger.

Finding 11:

Close to 90% of the upgrade failures are deterministic, not requiring any special timing to trigger.

Finding 12:

Half of upgrade bugs can be triggered by stress-testing operations with default configurations.

State of the art

Current distributed systems have some limitations:

  • They do not solve the problem of testing-workload generation.
  • There are also no mechanisms for their upgrade testing to systematically explore different version combinations, different configurations, different upgrade scenarios.

Testing and Detecting Upgrade Failures

According to our findings, we have designed

  • DUPTester to expose upgrade failures through in-house testing
  • DUPChecker to detect upgrade failures caused by data-syntax incompatibility.

DUPTester

Testing environment

DUPTester packages what the target system (e.g., Cassandra) is supposed to run on each physical node, with all its dependencies, in a container.

DUPTester simulates a 3-node cluster of the target distributed system on one single
physical machine using a container orchestration tool.

For every pair of versions to test, DUPTester systematically tests three upgrade scenarios:

  • Full-stop upgrade
  • Rolling upgrade
  • New node joining

Testing workload

DUPTester makes the best use of existing stress testing and unit testing.

Leveraging stress testing is straightforward. Test cases in stress testing consist of client-side commands to be issued to a cluster.

Leveraging unit testing is much harder. A unit test invokes internal functions of a system-under-test from a specially designed test harness and hence cannot be directly used as testing workload during system upgrade.

Therefore, DUPTester translate some unit tests into client-slide scripts that can be used directly as testing workload for three upgrade scenarios. At high level, DUPTester translator first constructs an abstract syntax tree (AST) for each unit test, and then synthesizes a Python program based on this AST.

Evaluation results

DUPTester is able to find 20 previously unknown (to our best knowledge) upgrade
failures in four distributed systems.

DUPTester experienced four false-positives. To evaluate false negatives, we apply DUPTester on 15 randomly sampled cases from our study, DUPTester can trigger 5 of them.

DUPChecker

we designed a static checker DUPChecker to search for two types of data-syntax incompatibility across versions:

  • (type-1) on data defined by serialization libraries
  • (type-2) on data of enum types

Data in serialization library

DUPChecker creates a parser for protocol files by extending the PyParsing module with the serialization library’s grammar, which then allows DUPChecker to compare the data format of the same data member from different versions and report any incompatible change of the first two categories as an error.

In total, DUPChecker found 878 incompatibilities.

We haven’t found any false-positive cases (by manually checking).

Data in enum type

DUPChecker first identifies the enum class whose member’s index has been written to a serialized output stream through data flow analysis.

Summary

I think this work stands out because it has conducted a comprehensive study and simplifies the case in software upgrade. Also, it leverages code synthesis and static analysis to build DUPTester and DUPChecker.
