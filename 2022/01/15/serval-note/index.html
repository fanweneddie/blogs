Note of Serval

Created2022-01-15Updated2022-01-15

This is the note of paper “Scaling symbolic evaluation for automated verification of systems code with Serval” by Luke Nelson et al. at SOSP’19.

Introduction

Serval is an extensible framework for writing automated verifiers.

Serval adopts recent advances in symbolic profiling, a systematic approach to identifying performance bottlenecks in symbolic evaluation.

Architecture

With Serval, developers implement a system using standard languages and tools, such as C and gcc. They write a specification to describe the intended behavior of the system in the Rosette language, which provides a decidable fragment of first-order logic. An automated verifier, also written in Rosette, reduces the semantics of the implementation code (e.g., in the form of machine instructions) into symbolic values through symbolic evaluation.

Figure 1: Architecture

There are two benefits:

  • It keeps the code and specification cleanly separated, which allows system developers to use standard system languages(C and assembly) and toolchains (gcc and binutils) for implementation.
  • Serval makes the proof steps fully automatic, requiring no code-level annotations such as loop invariants.

Symbolic execution

Rosette relies on two basic strategies: symbolic execution [27, 47] and
bounded model checking.

  • The former explores each path separately, which creates more opportunities for concrete evaluation but can lead to an exponential number of paths.

  • The latter merges the program state at each control-flow join, which creates compact encodings (polynomial in program size) but can lead to constraints that are difficult to solve.

To find the bottleneck, Rosette symbolic profiler produces an interactive web page.
The page shows statistics of symbolic evaluation for each function (e.g., the number of symbolic values, path splits, and state merges), and ranks function calls based on a score computed from these statistics to suggest likely bottlenecks.

Fixing bottlenecks usually requires domain knowledge not present in Rosette or the solver, such as the set of feasible values for a symbolic pc. Therefore, the verifier applies split-pc to the program counter.
