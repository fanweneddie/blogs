Note of Ctest

Created2022-01-15Updated2022-01-15

This is the note of paper “Testing Configuration Changes in Context to Prevent Production Failures” by Xudong Sun et al. at OSDI’20.

Introduction

Failures typically occur when

  • valid configuration changes expose dormant software bugs.
  • configuration changes violate undocumented, hidden configuration constraints.

However, existing techniques are limited in

  • Existing techniques only check configuration values and cannot detect configuration changes that cause code to fail.
  • Very few existing techniques can detect “legal misconfigurations” , which have syntactically and semantically valid values but result in unexpected behavior.

We present ctest, which can test configuration changes in the context of code that is affected by the changes. By connecting tests to production system configurations, ctests can effectively detect failure-inducing configurations.

Background

Checking configurations based on program behavior

A key capability of ctests is to check how actual configuration values impact program behavior.

In our experience, checking program behavior can be more effective in capturing failure-inducing configuration changes than checking configuration values against rules.

Detecting diverse misconfigurations

Such misconfig involves

  • custom regular expressions, user commands, and URIs.
  • invalid content referred to by path-related configurations.
  • violations of undocumented constraints that cannot be found by text-based document analysis.
  • dependencies among multiple configuration parameters.

Ctest Overview

Ctests detect both misconfigurations caused by assigning invalid values to configuration parameters and bugs in the code that are exposed by changing configuration parameters to new valid values.

Ctest Generation

Ctest generation proceeds in two steps.

  • First, the existing tests are parameterized by system configuration parameters,
    so that they can be run against different system configurations.
  • Second, the parameterized tests are transformed into ctests.

if ttt assumes specific values of a configuration parameter p∈Pp \in Pp∈P, ttt cannot be a ctest for ppp. But ttt can be a ctest for another independent parameter, say q∈Pq \in Pq∈P, if ttt does not assume a value for qqq.

Parameterization

The first step in generating ctests is to parameterize an existing
test ttt into t^\hat{t}t^.

Parameterization requires changing test code to read configuration values at
runtime, as provided by ctest infrastructure (§3.3), instead of from default configuration files or other test files.

We can do systematic parameterization by overwriting configuration values as the final step of configuration loading.

The parameterization step produces a parameterized test, t^(P)\hat{t}(\mathbb{P})t^(P), for each test ttt, where P\mathbb{P}P is the set of all system configuration parameters.

Transformation

A parameterized test t^(P)\hat{t}(\mathbb{P})t^(P) may not be a valid ctest—a ctest’s parameter set P^\hat{P}P^ should include only configuration parameters that can be checked by the ctest—the test logic and oracles should not assume specific parameter values.

It has two steps:

Parameter identification

We implemented both static and dynamic analysis.

Our static analysis taints the statements that can be reached by ttt and searches for configuration API usage among the tainted statement. However, since test code commonly passes configuration values to system code initialization, it is hard to precisely collect the exact configuration API usage in system code that may be reachable from tests. (why the size of parameter set is larger???)

We choose dynamic analysis under the assumption that most test cases are relatively deterministic. Our instrumentation inserts code to log the stack trace of each API invocation and the configuration parameter involved. With instrumentation in place, our dynamic analysis runs all existing tests and post-processes the log for each test ttt.

and excluding parameters that are reset from PPP.

Detecting implicit assumptions on configuration values

Not all parameter resets are performed using SET APIs. Some tests implicitly assume specific parameter values. We automatically identify and exclude the parameters on which tests have implicit assumptions(from P^\hat{P}P^).

Our intuition is that, if a test assumes specific values, then it will fail on different but valid values. So, a configuration parameter on which a test makes an implicit assumption can be identified by assigning a different valid value to the parameter and observing the outcome of the existing test.

Rewriting

In addition to the generated ctests, one can optionally manually rewrite an existing test to create a new ctest or rewrite a generated ctest to check more configuration parameters.

First, many configuration resets in test code are used for setting up test environment.

Second, some assertions in the test code assume the default configuration values, which can be safely removed or rewritten to the actual values being tested.
