Leveldb Iterator

Created2022-01-26Updated2022-01-26

This is the note and analysis of iterator in leveldb.

Overview

In STL, iterator is an abstraction of containers. Similarly, in leveldb, iterators are provided to access contents of a Table or a DB. With iterators, we can traverse contents and do compaction on leveldb.

The overview of iterators can be found in Figure 1 (copied from here).

Figure 1: Overview of Iterators

Introduction

Interface

This is the interface of an iterator and it is defined in file iterator.h.

Interface Iterator provides some methods:

  • Status as current position: Valid(), key(), value(), status()
  • Seeking: SeekToFirst(), SeekToLast(), Seek()
  • Traversing: Next(), Prev()

Here, Seek(target) seeks for the first element that is equal to or greater than target.

The child classes of Iterator are shown in Figure 2 (copied from here).

Figure 2: Hierarchy of Iterators

In Figure 2, Block::iter, MemTableIterator and LevelFileNumIterator are low-level iterator that accesses data directly in a data structure; TwoLevelIterator, MergingIterator and DBIter are high-level iterator that is an encapsulation of iterators.

Low-level

Low-level iterators gets data from a block, skiplist or vector. Therefore, their implementation depends on the data structures.

Block::iter

Block::iter reads blocks (including index blocks and data blocks, but excluding filter blocks) and it is defined in file block.cc.

The format of blogs is shown in Figure 3, where the iterator is at Entry 2. According to the given format, Block::iter has the following members:

  • The information of block, such as comparator_, data_, restarts_, num_restarts_ (In black in Figure 3).
  • The information of current entry, such as current_, restart_index_, key_, value and status (In red in Figure 3).

When the iterator has reached a new position, it will call DecodeEntry() to get key_ and value.

Seek() method uses binary search to find the target, since keys are sorted in each block.

When moving forward (Next()) or backward (Prev()), we need to adjust to the appropriate restart_point.

Figure 3: Block's Format

MemtableIterator

MemTableIterator read elements in MemTable and ImmemTable and it is defined in file memtable.cc. Actually, it is just an encapsulation of the iterator of skiplist.

The organization of skiplist is shown in Figure 4.

Figure 4: Organization of Skiplist

When the iterator of skiplist is seeking a target, it calls method FindGreaterOrEqual() to get the target. For example, in Figure 4, if we try to seek n7, it will go through (n1, 3) →\rightarrow→ (n1, 2) →\rightarrow→ (n4, 2) →\rightarrow→ (n6, 2) →\rightarrow→ (n6, 1) →\rightarrow→ (n6,0) →\rightarrow→ (n7, 0). Similarly, if the iterator is seeking a previous key of a target, it will find the last key that is smaller than the target.

With high probability, the time complexity of searching time is O(lg⁡n)O(\lg n)O(lgn), where nnn is the number of elements in the skiplist.

Version::LevelFileNumIterator

Version::LevelFileNumIterator is used to access a file in a level and it is defined in version_set.cc. Given a key target, we get the first file whose max key is equal to or greater than target. Since the metadata of those files are sorted in a vector (for level > 0), we can use a binary search to seek a key. The sorted vector is shown in Figure 5.

Figure 5: Level-file-num Iterator

High-level

High-level iterators encapsulates some iterators and they can be seen as a deeper abstraction.

TwoLevelIterator

TwoLevelIterator leverages iterators of index block and data block, and it is defined in file two_level_iterator.cc. When searching, it requires to seek first in index block, then in data block. So that’s why it is called “two-level” (As is shown in Figure 6).

Figure 6: Two-level Iterator

Here, the type of index block iterator and data block iterator is IteratorWrapper, because it can cache key and valid instead of calling method key() and valid() frequently (Since the invoke of virtual method has a relatively high cost).

MergingIterator

MergingIterator is used to do compaction and it is defined in file merger.cc. Basically, It manages some children iterators, and uses current_ to mark the iterator on current min/max key. Its format is shown in Figure 7.

Figure 7: Merging Iterator

When it calls method next(), it will find the smallest key next to current_'s. So this procedure is just the same as merging, and that’s why it is called MergingIterator.

DBIter

DBIter is defined in file db_iter.cc. This is the comment in that file, which shows the function of DBIter:

Basically, DBIter iterates over the whole database, including memtables and sstables. It internally uses MergingIterator.

DBIter has two directions. When moving forward, the internal iterator is at this->key(); When moving backward, the internal iterator is just before all entries whose user key = this->key().

Here are two important auxiliary methods: FindNextUserEntry() and FindPrevUserEntry().

  • FindNextUserEntry(): After the current key (inclusive), find the first key that is greater than the current key, or equal to the current key but with a smaller sequence number. If it has encounters a deleted key k, then the key whose user key is equal to k's and sequence number is smaller than k's will be ignored. Note that it should be ensured that the moving direction must be forward,
  • FindPrevUserEntry(): Before the current key (inclusive), find the previous key that is smaller than the current key and with a sequence that is smaller to the given sequence. It will ignore the deleted keys and store the right key and value into saved_key and saved_value. Note that it should be ensured that the moving direction must be backward.

With those two auxiliary methods, we have method Seek(): It first calls method Seek() of the internal iterator to locate the target, and then calls FindNextUserEntry() to skip the deleted and excessively new keys. Finally, it reaches the valid key that is bigger than the target and has a smaller sequence number than the given one.

Now we move on to method Prev() and Next(). For those methods, they need to move to a new key, which is different from method FindNextUserEntry() and FindPrevUserEntry(). Note that the operation of Prev() and Next() depends on the current direction:

  • For Prev(), we hope that its direction is backward, since it is more natural. If current direction is forward, the key is stored in iter_->key() instead of saved_key. Therefore, it need to go back and store the key in saved_key, which is ahead of iter->key().
  • For Next(), we hope that its direction is forward. If current direction is backward, the key is stored in saved_key instead of iter_->key(). Therefore, it need to go ahead and store the key in iter->key(), which is ahead of saved_key.

An example is shown in Figure 8, which is copied from here. In Figure 8, internal keys have the format of <user_key, sequence, type>. Suppose the given sequence is 2, current key is 4:2:1 and direction is forward.

Figure 8: Example of Next() and Prev() of DBIter

If we call Prev(), it will do the following things:

  • Firstly, it will call iter_->Prev() for two times and go to key 3:1:1. Its direction is changed to backward.
  • Secondly, it will call FindPrevUserEntry() and do the following things:
    • go to key 3:2:0 and find that it is deleted, so it just clear saved_key (which is 3:1:1).
    • skip key 3:3:1 since its sequence number > 2.
    • go to key 2:1:1 and save it into saved_key (since 2:1:1 > current saved_key = empty).
    • go to key 2:2:1 and save it into saved_key (since 2:2:1 > current saved_key = 2:1:1).
    • skip key 2:3:1 since its sequence number > 2.
    • terminate at key 1:1:1 since it is smaller than current saved_key = 2:2:1.
  • Now, iter->key() is 1:1:1 and saved_key is 2:2:1, and it finally returns saved_key.

If now we call Next(), it will do the following things:

  • Firstly, it will call iter_->Next() to go to 2:3:1. Its direction is changed to forward.
  • Secondly, it will call FindNextUserEntry() to go to 2:2:1.
  • Now, iter_->key() is 2:2:1 and saved_key is also 2:2:1, and it finally returns iter->key().
