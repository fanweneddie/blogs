A Study of Metastable Failures

Created2022-05-07Updated2022-05-07

This is the note of paper “Metastable Failures in Distributed Systems” by Bronson et. al at HotOS ’21.

Introduction

Metastable failures are the bad states that persist even when the trigger is removed. In those bad states, the goodput (i.e. throughput of useful work) is low. In order to leave metastable failures, we need to use some extremely forced methods such as reducing load.

The lifecycle of states and their relation can be shown in Figure 1. There are 3 states: Stable state can be self-healed but not efficient, and it can transform to vulnerable state as workload increases; Vulnerable state can work normally, but it may fall into metastable state by an unexpected trigger; Metastable state has a feedback loop to maintain its failure (just like deadlock), which has been described in paragraph above.

Figure 1: Lifecycle of States

Examples

Here, the authors introduce some examples of metastable failures, such as request retries, look-aside cache, slow error handling and link imbalance. In those cases, resource exhaustion cannot be alleviated by the given policy (designed for reliability or efficiency), so the feedback loop is formed.

Request Retries

Retrying failed request is common in web applications. For example, if some requests fail due to a short network outage, they will be resent in the next instant. Therefore, a spike of requests will be invoked, causing a high latency and a low throughput. And a similar situation will also happen next, which forms an feedback loop to sustain the congestion. When the number of requests is reduced, the system is recovered into a stable state.

This process is shown in Figure 2.

Figure 2: Process of Request Retries

Look-aside Caches

If cache miss happens in look-aside cache, the application will directly access data in the backing store and write it into cache (as is shown in Figure 3 from here).

Figure 3: Look-aside Cache

In a vulnerable state with many queries, if there exists some cache miss, then the database is more overloaded and timeout happens. Therefore, those queries are regarded as failed and cache cannot be updated. In this case, cache miss is persistent since it cannot be resolved by reading memory, which forms another feedback loop. When the number of queries is reduced, the system is recovered into a stable state.

This process is shown in Figure 4.

Figure 4: Process of Look-aside Cache

Slow Error Handling

For the ease of debugging (e.g. printing stack trace), the error handling code may have bad performance, which may make system remain in metastable state.

Link Imbalance

The authors show a real example of metastable failure in Facebook (Note that Figure 5~7 are all from this blog). In this example, the cache server has a connection pool to allocate connection to database on an aggregated link. The connection pool uses a MRU (most recently used) policy and the load of each link is evenly distributed in normal cases (as is shown in Figure 5).

Figure 5: Normal Case of the Connection Pool

However, when one link is congested (maybe due to network outage, hardware failure or load spike), the connections on that link is returned at last. Due to the MRU policy, those connections have the highest priority for allocation on next round (as is shown in Figure 6).

Figure 6: Congestion on a Link

Later, more queries will be send to that link. Even if the original congestion is solved, that link will encounter a new congestion because of the high workload (as is shown in Figure 7).

Figure 7: New Congestion on that Link

As we can see, the feedback loop of a metastable failure has formed. To recover from that, engineers can

  • change the routing algorithm (e.g. hashing function), or
  • change policy of connection allocation from MRU to LRU.

Therefore, the process of this scenario is shown in Figure 8.

Approaches

The authors describes the techniques to handle metastability. I just list out the main points of each approach, since I am not that familiar with every detailed example.

Changing Policy during Overload

In order to ensure a high goodput during overload, we can change routing or queueing policy during an overload. However, there are 2 challenges on this method:

  1. About the coordination, since it requires global information or communication to make a decision;
  2. To distinguish persistent overload from load spike. One heuristics is to check the minimum latency, since a high minimum latency is the sufficient condition of congestion.

Using priorities

For example, we can use a lower priority for a retried queries. However, what if those retried queries are critical for other concurrent queries (i.e. other concurrent queries depend on the retried queries)? In that case, the priority of retried queries should still be high.

There are 3 challenges on this method:

  1. Some priority systems may cause a high work amplification;
  2. Not every system can be implemented with a priority;
  3. Some system has implicit priorities that cannot be changed, which is the case of my question above.

Stress Tests

Stress tests can check whether metastable failures occur at a small scale. For a large scale, we need to check the running circumstances just like how Kraken tests the live traffic load. (I just read the abstract of Kraken, and I don’t know its detail.)

Organization Incentives

I don’t know what it means. Why a larger multiple of the threshold between stable and vulnerable states can exacerbate the feedback loop? I think that it can make recovery faster, since we only need to reduce less workload to reach stable state.

Fast Error Paths

We can simplify error handling by sending failures to a dedicated error logging thread instead of printing stack trace.

Outlier Hygiene

We can investigate the root cause of previous error, since it may also contribute to feedback loop or trigger of metastable failures.

Autoscaling

We can learn from elastic systems to maintain a capacity buffer to reduce being triggered and accelerate the recovery. However, reconfiguration will reduce capacity in a short term by performing state transfers and metadata updates.

Discussion

Goals

To handle the metastable failures, we have 2 separate goals:

  1. Efficient and free from metastable failures.
  2. Able to recover from metastable failures as soon as possible.

Those goals are difficult to realize, since it is not easy to detect vulnerable states and predict failures.

Like the example of Request Retries and Look-aside Caches above, we can develop techniques that can reduce load at metastable failure; Also, we can speed up the recovery by increasing the size of stable state.

To-do

In order to avoid metastable failures, we should

  • understand the reason of work amplification;
  • weaken the strongest feedback loop;
  • instead of doing small-scale reproduction of complex metastable failures, we can use a synthetic workload to prove the absence of a metastable failure;
  • understand how constant factors of feedback loops are affected by system changes.

In order to recover from metastable failures, we should

  • identify the metric that denotes feedback loop (e.g. latency and cache hit rate);
  • identify the range of safe values of the metric;
  • measure the boundary between the stable and vulnerable states (known as hidden capacity) by a stress test;
  • get the variation in the trigger size.
