CPAchecker Source Code Analysis

Created2022-04-12Updated2022-05-07

This is the source code analysis of CPAchecker.

Component

CPAMain

Connecting user with CPAchecker, CPAMain deals with the configuration.

CPAMain first parses the configuration from command line arguments into cpaConfig and initializes the tools such as CPAchecker, ResourceLimitChecker and reportGenerator. Them, it calls cpachecker.run(options.programs) to analyze the given program.

CPAchecker

CPAchecker is the main class to do program analysis, with the architecture shown in Figure 1 (from “CPAchecker: A Tool for Configurable Software Verification”).

Figure 1: Architecture of CPAchecker

In method run(), CPAchecker first initializes the CPA configuration and builds the CFA of the program. Then, it parses the specification from the spec file and uses CPABuilder to get the CPA (In our case, the CPA is the composition of LocationCPA and ControlAutomatonCPA). Later, it leverages CoreComponentsFactory to create the CPA algorithm based on the CFA and specification. After that, it calls method runAlgorithm() to run CPA algorithms by invoking run0() until a fixpoint has been reached.

To be specific, CPAchecker has the following important fields:

  • MainCPAStatistics stats: records the statistics of CPA, such as CPA create time.
  • Algorithm algorithm: the specific CPA algorithm.
  • ReachedSet reached: the set of reached state.
  • CFA cfa: abbreviated for control flow automaton, namely control flow graph. Each node represents a program location and each edge represents a statement of transfer.
  • Result result: the result of analysis.
  • Specification specification: about what we want to verify in specification file. In our case, the property is “Global Assert”.

Other Factories

CPAchecker leverages CPABuilder and CoreComponentsFactory. For CPABuilder, method buildCPAs() builds the hierarchy of CPA configuration according to CFA and specification; For CoreComponentsFactory, method createAlgorithm() creates the CPA algorithm.

Algorithm

ConfigurableProgramAnalysis

ConfigurableProgramAnalysis is an interface of a CPA, which contains

  • AbstractDomain: the domain of abstract states, which requires the definition of join() and isLessOrEqual(). It is an implementation of Lattice.
  • TransferRelation: the transfer relation is used to compute the successors of abstract states. Given a state and its edge in CFA, it can get all of its successors.
  • MergeOperator: the operator to merge newly-created abstract states with existing abstract states from the reached set.
  • StopOperator: the operator checking whether not to add the new abstract state.

Basically, there are tow kinds of CPAs: primitive CPA and composite CPA.

CPAAlgorithm

CPAAlgorithm is an encapsulation of ConfigurableProgramAnalysis.

run0() runs the CPA algorithm, whose pseudo code is shown in Figure 2 (from “Configurable Software Verification: Concretizing the Convergence of Model Checking and Program Analysis*”).

Some Classical CPAs

LocationCPA

Based on CFA, the abstract states of LocationCPA is program location and the transfer relation is the edge in CFA. Therefore, it marks the syntactic transfer relation.

ControlAutomatonCPA

ControlAutomatonCPA is an automaton CPA for assertion. Its automaton has 2 states: init and error, where false assertion means error state and states otherwise are init state. When we have reached an error state during CFA traversal, we stop the traversal and return false.
